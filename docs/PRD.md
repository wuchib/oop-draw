# OOP Draw MVP 产品需求文档

## 1. 产品背景与目标

### 1.1 产品背景
OOP Draw 是一个面向个人创作与轻量协作场景的通用白板应用，目标是提供类似 Excalidraw 的自由绘图体验，让用户能够以低门槛完成草图表达、流程讨论、结构示意、课堂讲解与截图分享等任务。

当前项目从 0 到 1 启动，首版不追求覆盖复杂团队协作与企业管理能力，而是优先解决以下核心问题：

- 用户能够快速打开一个空白画布并开始绘图。
- 用户能够用较自然的方式创建和编辑常见图形、文字、连线与图片。
- 用户能够在本地可靠地保存、恢复、导入和导出内容。
- 用户在 Web 端和桌面端获得一致的核心使用体验。

### 1.2 产品目标

#### 业务目标
- 在首版建立一个可独立使用的轻量白板产品基础能力。
- 形成后续扩展模板、分享、协作与账号体系的产品骨架。
- 为 Web 与桌面应用共用一套核心交互模型和文件模型。

#### 用户目标
- 3 秒内理解如何开始绘图。
- 10 分钟内完成一个简单图示、标注图或草图分享。
- 不依赖联网也能稳定完成编辑、保存与再次打开。

#### 产品原则
- 本地优先：优先保证本地编辑、自动保存和离线可用。
- 轻量优先：以自由表达为主，不做重型流程图工具。
- 一致优先：Web 桌面浏览器与桌面应用保持核心行为一致。
- 渐进增强：移动端首版支持浏览与基础编辑，不强求完整生产力。

### 1.3 成功指标
- 首次进入后，80% 以上新用户可在 30 秒内创建至少 1 个元素。
- 单用户在单画布中创建 500 个以内元素时，常规缩放、拖拽、框选保持可用且无明显卡顿。
- 自动保存成功率不低于 99%。
- 本地草稿恢复成功率不低于 95%。

## 2. 用户角色与核心场景

### 2.1 目标用户

#### 个人创作者
需要快速记录想法、草拟结构图、为文档或演示制作配图的用户。

#### 产品与研发人员
需要快速绘制流程、模块关系、交互草图和讨论图示的用户。

#### 教师与讲解者
需要在讲解过程中手写标注、插入图片、绘制示意关系并导出的用户。

#### 学生与知识工作者
需要整理笔记、做结构化图解、进行轻量知识表达的用户。

### 2.2 核心使用场景
- 用户打开应用后直接在空白画布上画框、写字、连线，快速表达思路。
- 用户导入一张图片，在其上进行圈选、箭头标注和文字说明。
- 用户编辑中的画布被意外关闭后，下次进入应用可以恢复草稿。
- 用户将画布导出为 PNG 或 SVG，用于文档、聊天工具、演示文稿或课程资料。
- 用户在桌面应用中打开本地画布文件，继续进行编辑和再次保存。

## 3. MVP 范围与非范围

### 3.1 MVP 范围
首版 MVP 包含以下能力：

- 无限画布与基础视图控制。
- 常用图形、箭头、自由绘制、文本、图片等绘图能力。
- 选择、移动、缩放、组合、层级管理等基础编辑能力。
- 撤销重做、快捷键、样式设置、对齐吸附等提升效率的编辑体验。
- 新建、打开、自动保存、本地草稿恢复、导入导出 PNG/SVG/JSON。
- Web 端与桌面端一致的核心操作体验。
- 移动端浏览、选择与基础编辑能力。

### 3.2 非范围
以下内容明确不进入首版 MVP：

- 多人实时协作。
- 在线评论与批注会话。
- 团队空间、角色权限和组织管理。
- 模板市场与模板推荐系统。
- 云端账号体系与跨设备云同步。
- 复杂流程图自动布局和高级图表组件。

### 3.3 延后原因
- 实时协作与账号体系会显著增加同步、冲突处理和服务端复杂度，不适合首版。
- 模板市场和团队空间依赖稳定的基础编辑体验与用户体系，应在核心绘图能力成熟后推进。
- 自动布局和高级图表会改变产品定位，偏离“轻量自由白板”的首版目标。

## 4. 功能需求

### 4.1 模块一：画布与视图

#### 4.1.1 目标
让用户可以在一个无明显边界的空间中进行自由创作，并以低认知成本进行浏览、定位和聚焦。

#### 4.1.2 功能项

##### 无限画布
- 系统应提供可持续扩展的画布区域，用户不需要预先设置画布尺寸才能开始绘图。
- 初始进入时，视图聚焦于默认中心区域。

##### 缩放与平移
- 支持鼠标滚轮缩放、触控板缩放、拖拽平移。
- 支持通过工具栏和快捷键进行放大、缩小、重置缩放。
- 缩放应以当前指针附近或视图中心为主要参考点，避免明显跳变。

##### 网格与对齐参考
- 支持网格显示开关。
- 支持元素移动过程中的吸附对齐提示。
- 支持对齐参考线展示，帮助用户发现边缘、中心与间距关系。

##### 缩略导航
- 提供可选的缩略导航视图，用于快速定位当前画布区域。
- 当画布内容较多时，缩略导航不应明显遮挡主要编辑区域。

##### 主题
- 支持浅色主题与深色主题。
- 主题切换后不改变图形本身语义，仅改变应用壳层与辅助视觉表现。

#### 4.1.3 核心规则
- 画布默认无分页概念。
- 视图状态应作为画布文件的一部分可被恢复。
- 网格和缩略导航允许被关闭，但关闭后不影响核心编辑流程。

#### 4.1.4 优先级
- P0：无限画布、缩放、平移。
- P1：网格、吸附提示。
- P1：缩略导航、主题。

#### 4.1.5 验收标准
- 用户进入应用后无需任何配置即可开始绘图。
- 用户可通过至少两种方式完成缩放。
- 用户在元素移动时可看到对齐辅助信息。

### 4.2 模块二：绘图能力

#### 4.2.1 目标
支持用户用最少步骤完成大多数轻量示意图与标注图表达。

#### 4.2.2 功能项

##### 基础图形
- 支持矩形、菱形、椭圆。
- 图形创建时支持拖拽决定尺寸。
- 图形创建后支持再次选中并编辑大小、位置和样式。

##### 线条与箭头
- 支持直线、单向箭头。
- 连线应支持端点拖动调整。
- 连线样式至少支持颜色、粗细、虚实变化。

##### 自由绘制
- 支持铅笔或自由手绘工具。
- 支持连续笔迹绘制与单笔完成。

##### 文本
- 支持点击创建文本和在图形附近添加文本。
- 支持文本内容编辑、字体大小、颜色与基础对齐能力。

##### 图片
- 支持导入本地图片到画布。
- 图片支持移动、缩放、删除和层级调整。

##### 选择与框选
- 支持单选、多选、拖拽框选。
- 支持对多个元素进行整体移动和样式批量修改。

##### 复制粘贴
- 支持键盘复制、粘贴、剪切。
- 支持跨画布会话内复制粘贴基础元素。

##### 层级与组合
- 支持置顶、置底、上移一层、下移一层。
- 支持组合与取消组合。

#### 4.2.3 核心规则
- 首版不要求复杂连接器自动吸附到图形锚点，但要保留后续扩展空间。
- 图片作为画布资源的一部分进行引用管理。
- 文本编辑完成后应回到选择或上一个工具状态，避免卡在不可预期模式。

#### 4.2.4 优先级
- P0：矩形、菱形、椭圆、直线、箭头、自由绘制、文本、图片。
- P0：选择、框选、复制粘贴。
- P1：层级管理、组合。

#### 4.2.5 验收标准
- 用户可独立完成一个包含图形、箭头、文本和图片的简单示意图。
- 用户可对多个元素同时移动并修改至少一种公共样式。
- 用户导入图片后可再次编辑位置和大小。

### 4.3 模块三：编辑体验

#### 4.3.1 目标
降低高频操作成本，保证编辑过程稳定、可撤回、可恢复。

#### 4.3.2 功能项

##### 撤销重做
- 支持至少 50 步操作历史。
- 支持快捷键触发撤销和重做。

##### 快捷键
- 提供工具切换、删除、复制粘贴、撤销重做、缩放相关快捷键。
- 桌面端与 Web 桌面浏览器保持主要快捷键一致。

##### 吸附与对齐
- 在移动和缩放元素时提供吸附与参考线辅助。
- 支持基本的中心对齐、边缘对齐感知。

##### 样式设置
- 支持设置线条颜色、填充颜色、粗细、透明度、文本大小等基础样式。
- 当多选元素存在共同样式项时，支持批量修改。

##### 锁定
- 支持锁定元素，防止误拖动和误修改。
- 锁定元素仍可被查看和通过图层能力进行管理。

##### 删除恢复策略
- 支持删除所选元素。
- 删除后可通过撤销恢复。

#### 4.3.3 核心规则
- 所有核心编辑动作都必须进入撤销栈。
- 快捷键不应与系统高频保留快捷键产生严重冲突。
- 被锁定元素默认不可直接拖动，但允许解锁后继续编辑。

#### 4.3.4 优先级
- P0：撤销重做、删除、快捷键、基础样式设置。
- P1：吸附与对齐、锁定、多选批量样式。

#### 4.3.5 验收标准
- 用户误删元素后可在一轮操作内通过撤销恢复。
- 主要绘图与编辑动作均可通过快捷键辅助完成。
- 多选元素时可统一调整至少颜色和线宽。

### 4.4 模块四：文件能力

#### 4.4.1 目标
保证用户内容可保存、可恢复、可迁移、可分享。

#### 4.4.2 功能项

##### 新建与打开
- 支持创建空白画布。
- 支持打开本地画布文件继续编辑。

##### 自动保存
- 编辑过程中自动保存草稿。
- 自动保存应尽量避免打断用户。

##### 本地草稿恢复
- 应用异常关闭、标签页关闭或系统中断后，再次进入应提示恢复最近草稿。
- 恢复流程应明确展示来源和时间，避免误覆盖。

##### 导入
- 支持导入历史画布文件。
- 支持导入图片文件到当前画布。

##### 导出
- 支持导出 PNG、SVG、JSON。
- 导出时允许选择导出全部内容或当前选中内容。

##### 桌面端文件关联预留
- 在 PRD 中预留桌面端文件双击打开能力，但不要求首版必须完成。

#### 4.4.3 画布文件对象
产品层的画布文件至少包含以下内容：

- 元素集合。
- 元素样式信息。
- 视图状态。
- 资源引用信息。
- 文件版本号。

#### 4.4.4 核心规则
- JSON 作为首版可编辑原始格式。
- PNG 和 SVG 作为对外交付格式，不要求可逆完整恢复编辑状态。
- 自动保存与手动保存要有明确区别，避免用户误以为已生成正式文件。

#### 4.4.5 优先级
- P0：新建、打开、自动保存、草稿恢复、导入图片、导入导出 JSON/PNG/SVG。
- P2：桌面端文件关联。

#### 4.4.6 验收标准
- 用户可新建画布并保存为本地文件，再次打开后继续编辑。
- 用户在异常关闭后可恢复最近编辑内容。
- 用户可导出 PNG 和 SVG 并在常见查看工具中打开。

### 4.5 模块五：分享能力

#### 4.5.1 目标
在不引入复杂服务端的前提下，满足用户将作品交付给他人查看的基本需求。

#### 4.5.2 功能项

##### 文件分享
- 用户可通过导出 PNG、SVG、JSON 进行分享。
- 导出流程中应清晰说明不同格式适用场景。

##### 只读分享占位
- PRD 中定义后续支持“只读分享链接”这一产品方向。
- 首版 UI 可预留入口或文案说明，但不要求实际生成在线链接。

#### 4.5.3 核心规则
- 首版不依赖服务端即可完成主要分享诉求。
- 若产品界面出现“分享”入口，必须明确区分“导出分享”和“在线链接分享”状态，避免误导。

#### 4.5.4 优先级
- P0：文件导出分享。
- P2：只读链接分享占位。

#### 4.5.5 验收标准
- 用户无需登录即可将画布导出并发送给他人查看。
- 首版界面不应让用户误以为支持多人在线编辑。

### 4.6 模块六：跨端能力

#### 4.6.1 目标
在不同终端提供统一的产品认知与可用体验，同时根据终端特点进行能力分级。

#### 4.6.2 平台支持矩阵

| 平台 | 支持级别 | 说明 |
| --- | --- | --- |
| Web 桌面浏览器 | 完整支持 | 作为首版核心编辑平台 |
| Web 移动浏览器 | 基础支持 | 支持浏览、选择、基础编辑 |
| 桌面应用 | 完整支持 | 与 Web 核心编辑能力一致 |

#### 4.6.3 核心规则
- Web 桌面浏览器是首版体验基准。
- 桌面应用不新增独占绘图能力，但可增强文件打开和本地集成体验。
- 移动端不要求完整快捷键体系，不要求所有批量操作都高效可用。

#### 4.6.4 优先级
- P0：Web 桌面浏览器。
- P1：桌面应用。
- P1：移动端浏览与基础编辑。

#### 4.6.5 验收标准
- Web 桌面浏览器和桌面应用在核心工具集上保持一致。
- 移动端可打开现有画布并完成基础元素选择、移动和简单编辑。

## 5. 关键交互流程

### 5.1 首次进入并开始绘图
1. 用户打开应用。
2. 系统展示空白画布、默认工具栏和基础提示。
3. 用户选择图形或直接使用默认工具开始绘制。
4. 用户完成初次绘制后，系统进入正常编辑状态，并开始自动保存草稿。

### 5.2 保存与恢复未完成画布
1. 用户在画布中持续编辑。
2. 系统在后台自动保存本地草稿。
3. 用户异常退出或关闭页面。
4. 用户再次进入应用时，系统检测到未完成草稿。
5. 系统提示用户恢复最近草稿或放弃草稿。
6. 用户选择恢复后回到上次编辑状态。

### 5.3 导入图片或历史文件再编辑
1. 用户选择导入图片或打开历史画布文件。
2. 系统校验文件类型并加载资源。
3. 图片进入当前画布，或历史画布整体恢复。
4. 用户继续移动、缩放、标注和保存。

### 5.4 导出分享给他人查看
1. 用户完成内容编辑。
2. 用户点击导出。
3. 系统展示 PNG、SVG、JSON 等导出选项及适用说明。
4. 用户选择导出格式与范围。
5. 系统生成导出文件并交付给用户。
6. 用户可通过聊天、邮件、文档等方式分享文件。

### 5.5 桌面端打开本地文件继续编辑
1. 用户在桌面应用中打开本地画布文件。
2. 系统加载元素、样式、视图状态与资源引用。
3. 用户继续编辑并再次保存。
4. 系统保证内容版本兼容，不因二次打开而丢失主要信息。

## 6. 非功能需求

### 6.1 性能要求
- 冷启动后首屏可交互时间目标小于 3 秒，适用于常见桌面浏览器环境。
- 单画布 500 个以内元素时，常规拖拽、缩放、框选应保持流畅可用。
- 导出 PNG/SVG 的等待时间在常见中小型画布下应控制在用户可接受范围内，不出现长时间无反馈。

### 6.2 可靠性要求
- 自动保存应在主要编辑动作后稳定触发，不因单次失败导致内容整体丢失。
- 本地草稿恢复应具备明确的错误提示和兜底策略。
- 文件打开失败时，应提示原因，不得直接崩溃或清空当前会话。

### 6.3 离线可用性
- 首版核心编辑、保存、导入导出能力应支持离线使用。
- 无网络时不影响本地草稿恢复和本地文件打开。

### 6.4 数据安全与隐私
- 首版默认不依赖账号，不主动上传用户画布内容到云端。
- 本地缓存与草稿需有明确生命周期与覆盖策略，避免用户不知情丢失。
- 若后续引入在线分享，必须单独定义权限、安全与隐私策略。

### 6.5 兼容性要求
- Web 端优先支持主流 Chromium 浏览器和最新版 Safari、Firefox。
- 桌面端需支持主流桌面操作系统的基础文件读写与窗口使用场景。
- 移动端以主流手机浏览器为目标，优先保证查看和基础操作。

### 6.6 可观测性与错误恢复
- 首版应记录关键失败事件，如导出失败、文件解析失败、自动保存失败。
- 用户遇到失败时，应看到可理解的提示，而不是技术性报错堆栈。
- 对于可恢复错误，系统应提供重试或回退路径。

## 7. 版本规划与里程碑

### 7.1 V1 MVP
目标：形成可独立交付的单人白板应用。

范围：
- 无限画布与基础视图控制。
- 常用绘图工具与编辑能力。
- 自动保存、本地草稿恢复、本地文件打开与导出。
- Web 桌面浏览器完整支持。
- 桌面应用和移动端基础版本。

## 8. 风险与待确认项

### 8.1 主要风险
- 画布性能与元素渲染模型若设计不当，会影响后续扩展能力。
- 图片资源管理若缺少统一策略，可能导致文件膨胀和导入导出不稳定。
- Web 与桌面双端并进会增加测试矩阵和交付复杂度。
- 移动端基础编辑若设计过重，可能拖慢首版交付节奏。

### 8.2 待确认项
- 桌面应用技术路线是否采用 Electron、Tauri 或其他方案。
- 首版是否提供显式“保存为文件”与“仅保留本地草稿”两套入口。
- 导出 PNG/SVG 时是否支持背景透明、边距裁剪和分辨率选项。
- 移动端基础编辑范围是否包含文本编辑与图片导入。
- 首版是否需要提供示例画布或新手引导。
